# TUR GUI 0.3.0 Plan

## Purpose

Turn `tur-tauri` from a single-window MVP shell into a real desktop download manager shape:

- a persistent app process with tray behavior
- a main list window for overview
- a per-download instance window for focused control
- tighter IDM-inspired ergonomics without copying IDM literally

This release should also finish the meaningful unfinished carry-over work from `0.2.0` where that work is a prerequisite for the new window model.

---

## Current Position

What is already true:

- `tur-rs` is the real engine
- the GUI already drives `TurService` directly
- session-aware inputs exist
- worker diagnostics exist
- protocol visibility now exists
- pause/resume/cancel is substantially healthier than before

What is not yet in the right product shape:

- the app still behaves too much like one broad MVP surface
- the download instance flow is not yet first-class
- tray/minimize-to-background behavior is not yet the real product model
- the list/detail hierarchy is not yet aligned with classic download-manager UX

---

## Product Direction

Use IDM as **interaction inspiration**, not as a pixel-copy target.

The target feel is:

- lightweight
- clear
- dense where useful
- fast to operate
- modernized
- not over-animated
- not visually cheap

Theme support:

- support both light and dark themes
- keep both modes first-class
- avoid a giant theming system in this release

---

## Release Goal

Ship the first real desktop UX shape for `tur`:

- a smaller, cleaner main window for task overview
- a dedicated per-download window for active task control
- tray-first application lifecycle
- single-instance app with multiple managed windows
- enough information density to feel like a real download manager

---

## Non-Goals for 0.3.0

- browser extension integration
- OS shell integration for text files / links beyond placeholder hooks
- scheduler power-user control panels
- heavy motion design
- large settings system
- download categorization / rules engine
- media-sniffing / browser capture

---

## Important Notes

### 1. `0.2.0` carry-over is real

If parts of `0.2.0` are still unfinished, finish the parts that are directly required for `0.3.0` instead of pretending they are already closed.

In practice that means:

- runtime validation truth
- task hygiene / row lifecycle
- file/folder actions
- better task management

must be handled during `0.3.0` if still missing.

### 2. Single instance, multiple windows

The app remains a **single application instance**, but it can own multiple windows:

- one main window
- one per-download instance window per active download flow

### 3. Tray-first behavior

Closing the main window should normally hide the app to tray rather than fully terminate it.

“Quit” should be an explicit tray action or menu action.

### 4. Lightweight matters

Do not build a giant reactive dashboard for every frame.

Heavy per-connection rendering should only appear when the detail panel is expanded.

### 5. `tur-rs 0.9.1` is a dependency for “excellent” pause/resume UX

`0.3.0` can and should proceed with the GUI/window model now, but it should not pretend to solve deep engine lifecycle issues in the frontend.

In particular:

- protocol display can rely on the current `tur-rs` event surface
- worker diagnostics can rely on the current `tur-rs` event surface
- but best-in-class pause/resume behavior depends on `tur-rs/tasks/0.9.1.md`

So:

- GUI work may proceed in parallel
- engine hibernation/resume hardening belongs in `tur-rs`, not in Tauri glue code

## Phase 1 — Finish the Real 0.2.0 Carry-Over

### Goal

Do not build the IDM-style UX on top of unfinished task lifecycle behavior.

### Work

- verify task persistence / restore status honestly
- add/finish:
  - remove row
  - clear completed
  - clear failed
  - retry failed
- add file/folder actions if still missing:
  - reveal folder
  - open file where appropriate
- tighten runtime validation notes from the MVP phase

### Success criteria

- the current task list is trustworthy before we split behavior across windows

---

## Phase 2 — Window Model and App Lifecycle

### Goal

Define the real app model.

### Work

- keep single-instance app behavior
- keep multiple managed windows
- define window roles:
  - `main`
  - `download-instance:<task-id>`
- tray lifecycle:
  - normal window close hides to tray
  - tray menu includes:
    - show main window
    - new download
    - quit
- ensure idle tray state is lightweight
- confirm app shutdown is explicit and safe

### Success criteria

- the app behaves like a persistent desktop downloader, not just a page in a shell

---

## Phase 3 — Main Window Redesign

### Goal

Make the main window an overview/control surface, not the place where every detail lives.

### Target Shape

- slightly smaller than the current oversized layout
- roomy but not wasteful
- overview-first

### Main Window Responsibilities

- list downloads
- show aggregate/top-level state
- let user open or focus a download-instance window
- allow quick actions where appropriate

### Information to show in the list

Per task:

- filename
- status badge
  - in-progress = blue
  - completed = green
  - remaining/paused/stopped = yellow/amber family
  - failed = red
- current speed
- remaining bytes / progress
- protocol
- location summary

Top-level:

- total speed
- active download count
- paused/stopped count

### Success criteria

- the main window stays readable with many tasks and does not try to be the full inspector

---

## Phase 4 — Download Instance Window

### Goal

Create the focused per-download window inspired by the classic IDM interaction model.

### Target Shape

Base:

- roughly `728 x 428`

With details expanded:

- height increases meaningfully, roughly “double-pane” behavior
- not a huge freeform resize explosion

### Responsibilities

- show one task clearly
- support:
  - pause
  - resume
  - cancel/stop
  - retry where applicable
- show important live info without noise

### Base visible info

- file name
- source URL/origin
- save location
- status
- speed
- received / total
- remaining
- ETA
- protocol
- active connection count

### Detail panel

Collapsed by default or compact by default based on best UX during implementation.

When opened:

- show per-connection rows
- each row should include the most useful available signals:
  - connection id
  - state
  - speed
  - transferred bytes
  - assigned range / range progress
  - retry/detail note if relevant

Important:

- render this only when the detail panel is open
- do not make every row expensive all the time

### Success criteria

- the instance window is the best place to understand and control one download

---

## Phase 5 — Connection Diagnostics UX

### Goal

Use the `tur-rs` worker diagnostics well instead of dumping them raw.

### Work

- map worker states to clear user-facing labels
- distinguish:
  - connecting
  - downloading
  - waiting
  - retrying
  - paused
  - finished
  - stopped
- make worker rows compact and legible
- ensure connection counts do not feel misleading
- if the engine exposes protocol and worker diagnostics together, show them coherently

### Success criteria

- advanced users can understand what the downloader is doing without the UI feeling like a debug console

---

## Phase 6 — Light/Dark Theme Support

### Goal

Support both dark and light themes without ballooning scope.

### Work

- define a small tokenized color system
- build both dark and light variants
- keep contrast strong enough for dense information panels
- preserve the same layout and hierarchy across themes

### Success criteria

- the app looks intentional in both themes, not “dark-first with a broken light mode”

---

## Phase 7 — New Download Flow

### Goal

Make starting a new download feel product-grade.

### Work

- new download entry from:
  - main window
  - tray menu
- per-download instance window can act as the main place to confirm:
  - URL(s)
  - target location
  - optional session fields
- support link-list input cleanly
- support folder selection clearly

### Success criteria

- a new task feels like a focused flow, not a form crammed into the dashboard

---

## Phase 8 — Tray and Background Idling

### Goal

Move toward the “download manager always available, but lightweight” model.

### Work

- close-to-tray behavior
- restore-from-tray behavior
- ensure idle state is not doing pointless heavy polling
- document platform differences/caveats

### Success criteria

- the app can stay around like a real desktop utility without feeling heavy or annoying

---

## Phase 9 — Packaging and Platform Truth

### Goal

Treat this like a cross-platform desktop app, not only a Linux dev setup.

### Work

- verify Linux local behavior
- ensure Windows/macOS compile/package status is known where practical
- validate tray behavior by platform
- validate icon/bundle metadata

### Success criteria

- the product direction is honest across Linux/macOS/Windows

---

## Suggested Implementation Order

1. Finish meaningful `0.2.0` carry-over
2. Window model and lifecycle
3. Main window redesign
4. Download instance window
5. Connection diagnostics UX
6. Theme support
7. New download flow
8. Tray/background behavior
9. Packaging truth

---

## Validation Requirements

Before calling `0.3.0` done, verify:

- one real download through the new instance window flow
- one pause/resume cycle from the instance window
- one cancel/stop path
- main window list stays coherent while instance windows are open/closed
- tray hide/show/quit works on the local platform
- connection detail panel expands/collapses correctly
- protocol display matches observed download behavior

---

## Deliverables

At the end of `0.3.0`, `tur-tauri` should have:

- a real main window
- a real per-download instance window
- a tray-first app model
- usable connection diagnostics
- light/dark support
- a cleaner IDM-inspired interaction model

without turning into a bloated UI experiment.
