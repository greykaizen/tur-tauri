# TUR GUI 0.1.0 Plan

## Purpose

Create the first usable desktop GUI for `tur` as a separate Tauri project
that depends on `tur-rs`.

This is an MVP release, not a feature-complete desktop client.

The goal is to prove the product shape:

- `tur-rs` owns downloader logic
- `tur` GUI owns desktop UX
- the GUI talks to the `TurService` facade instead of shelling out to the CLI

---

## Release Goal

Ship a cross-platform desktop MVP for Linux/macOS/Windows that can:

- add downloads
- choose output directory
- display progress, total size, speed, and status
- pause / resume / cancel
- survive ordinary task churn cleanly

This release should feel like a real downloader, even if it is intentionally minimal.

---

## Non-Goals for 0.1.0

- advanced scheduler tuning controls
- browser/webview session extraction
- CAPTCHA/challenge solving
- deep settings surface
- polished theming/branding perfection
- multi-window complexity
- system tray background daemon mode

---

## Architecture

### Dependency direction

- `tur` GUI -> `tur-rs`

The Tauri backend should depend on `tur-rs` directly during development
with `default-features = false` so the GUI does not pull TUI-only bloat.

The release dependency can switch to the published crate version once the
GUI backend is stable.

### Integration model

Use `TurService` from `tur-rs` inside the Tauri backend:

- one service instance per app session
- backend command handlers submit/control downloads
- backend emits normalized progress events to the frontend

Do **not** shell out to the `tur` binary for the MVP.

### Frontend foundation

Use:

- Tailwind CSS for the initial UI foundation
- Bun as the frontend package manager/runtime
- Tauri dialog plugin for directory picking
- Tauri store plugin for lightweight local settings
- Tauri notification plugin for completion/failure feedback
- Tauri single-instance plugin so a second launch focuses the existing app
- Tauri autostart plugin for opt-in startup behavior
- Tauri fs plugin where frontend-side file access becomes necessary

Do **not** add system tray or background daemon behavior in `0.1.0`.

---

## Phase 1 — Backend Wiring

### Goal

Make the Tauri backend own a live `TurService`.

### Work

- add `tur-rs` as a path dependency in `src-tauri/Cargo.toml`
- create backend state holding:
  - `TurService`
  - task registry
  - event fanout to the frontend
- define backend commands for:
  - `start_download`
  - `pause_download`
  - `resume_download`
  - `cancel_download`
  - `list_downloads`

### Success criteria

- the GUI can start and control real downloads through `tur-rs`

---

## Phase 2 — Event Bridge

### Goal

Stream `tur-rs` download updates into the frontend cleanly.

### Work

- define a frontend-safe event payload model
- translate `DownloadUpdate` into Tauri events
- include:
  - task id
  - filename / url
  - downloaded bytes
  - total size
  - speed
  - status
  - error string when present

### Success criteria

- the frontend can subscribe once and reactively update the UI

---

## Phase 3 — Basic UI

### Goal

Build the smallest useful downloader UI.

### Work

- input for URL
- support one or more URLs
- output directory picker/input
- import links from a local text file
- add-download button
- task list/table
- selected-task controls:
  - pause
  - resume
  - cancel
- visible status badges
- progress bars
- speed/size text

### Success criteria

- a normal user can download files without touching the terminal

---

## Phase 4 — State Model

### Goal

Keep the frontend state simple and durable enough for the MVP.

### Work

- maintain an in-memory download store keyed by task id
- update the store from Tauri events
- keep command state transitions explicit
- handle app startup/shutdown without UI crashes

### Success criteria

- task rows stay coherent through progress/status churn

---

## Phase 5 — Error Handling

### Goal

Make failures understandable.

### Work

- display `DownloadStatus::Error` clearly
- surface challenge/session-aware failures from `tur-rs`
- show when a download was paused/cancelled/completed

### Success criteria

- users can tell the difference between:
  - network failure
  - challenge/interstitial failure
  - user cancellation

---

## Phase 6 — Session-Aware Inputs (Minimal)

### Goal

Support the minimum real-world context knobs without cluttering the GUI.

### Work

For the MVP, support optional advanced fields in a collapsible area:

- Referer
- Bearer token
- Cookie file path
- custom headers (optional simple list)

These should map directly to `tur-rs` request/session context.

### Success criteria

- the GUI can drive the new `0.8.x` session-aware core when needed

---

## Phase 7 — Validation

### Goal

Prove the MVP works on the supported desktop targets.

### Work

- local run on Linux
- compile verification for macOS/Windows where practical
- one real file download through the GUI path
- one pause/resume path
- one cancel path
- one session-aware path if available

### Success criteria

- MVP is actually usable, not just scaffolded

---

## Suggested Order

1. Backend wiring
2. Event bridge
3. Basic UI
4. State model
5. Error handling
6. Minimal session-aware inputs
7. Validation

---

## Final Assessment

`0.1.0` should prove that `tur` as a GUI product is viable without overreaching.

If done correctly, this gives you:

- a usable desktop downloader MVP
- a clean separation from `tur-rs`
- a frontend foundation that can grow without contaminating the downloader core
